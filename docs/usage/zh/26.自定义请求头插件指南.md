# Snow CLI 使用文档——自定义请求头插件指南

## 概述

Snow CLI 的自定义请求头功能允许你为 API 请求添加额外的 HTTP 头。从本版本开始，请求头的**值**支持 `{{占位符}}` 语法，可以在每次 API 请求时由插件动态解析。

这意味着你可以：

- 在请求头中引用动态生成的 OAuth Token，无需手动更新
- 注入时间戳、请求 ID 等运行时变量
- 从密钥管理服务（如 Vault、AWS Secrets Manager）读取凭据
- 根据当前环境（开发/生产）注入不同的值

> 如果不安装任何插件，自定义请求头的行为与之前完全一致——`{{占位符}}` 会原样发送，不会产生任何额外开销。

## 插件目录

Snow CLI 从以下目录加载自定义请求头插件：

```bash
~/.snow/plugin/custom_headers/
```

支持的文件扩展名：

- `.js`
- `.mjs`（推荐，纯 ES Module 写法）
- `.cjs`

说明：

- 只支持从用户目录加载插件。
- Snow CLI 按文件名排序后，在首次遇到包含 `{{占位符}}` 的请求头时懒加载所有插件。
- 新增、修改或删除插件文件后会自动热重载，无需重启 Snow CLI。
- 如果请求头的值中不包含任何 `{{占位符}}`，插件不会被加载，行为与原来完全一致。

## 支持的导出形式

一个插件模块可以使用以下任意一种导出形式（加载器会扫描所有形式）：

```js
export default { ... }
```

```js
export const customHeaderPlugin = { ... }
```

```js
export const customHeaderPlugins = [{ ... }, { ... }]
```

如果多个插件都能解析同一个占位符，按文件名排序后先加载的插件优先（first-wins）。

## 插件结构

每个插件都必须满足以下结构（使用 TypeScript 表达更直观，但插件文件本身是普通 JavaScript）：

```ts
interface CustomHeaderPlugin {
	id: string; // 唯一标识
	name?: string; // 可选，显示名（用于日志）
	enable?: boolean; // 可选，默认 true
	resolve(
		placeholders: string[], // 当前请求头中出现的所有占位符名称
		context: CustomHeaderPluginContext, // 运行时上下文
	): Promise<Record<string, string>>; // 返回占位符名 → 解析值的映射
}

interface CustomHeaderPluginContext {
	cwd: string; // 当前工作目录
	platform: string; // 操作系统平台（如 'darwin'、'win32'、'linux'）
}
```

字段说明：

- `id`：插件的唯一标识，用于日志输出。一旦发布请保持稳定。
- `name`（可选）：人类可读的名称，仅用于日志。
- `enable`（可选）：默认 `true`。设为 `false` 可临时停用插件而无需删除文件。
- `resolve(placeholders, context)`：核心解析函数。
  - `placeholders`：从当前激活的请求头方案中提取的所有唯一占位符名称。例如请求头值 `Bearer {{token}}` 会传入 `["token"]`。
  - `context`：运行时上下文，包含 `cwd` 和 `platform`。
  - 返回值：一个 `Record<string, string>`，键是占位符名称，值是解析后的字符串。**只需返回该插件成功解析的占位符**，无法解析的占位符直接省略即可——它们可能由其他插件解析，或保持原样。

## 占位符语法

在自定义请求头的**值**中使用 `{{占位符名称}}` 语法：

| 请求头键        | 请求头值               | 提取的占位符                 |
| --------------- | ---------------------- | ---------------------------- |
| `Authorization` | `Bearer {{token}}`     | `["token"]`                  |
| `X-Timestamp`   | `{{timestamp}}`        | `["timestamp"]`              |
| `X-Request-ID`  | `req-{{uuid}}-{{env}}` | `["uuid", "env"]`            |
| `X-API-Key`     | `sk-abc123`            | `[]`（无占位符，不触发插件） |

规则：

- 占位符名称会自动 `trim()`，所以 `{{ token }}` 和 `{{token}}` 等价。
- 同一个值中可以包含多个占位符。
- 同一个占位符名称在多个请求头中出现时，只解析一次，结果复用到所有位置。
- 没有被任何插件解析的占位符保持原样（例如 `{{token}}` 会被原样发送）。

## 解析流程

```
请求头值包含 {{占位符}}？
├── 否 → 直接返回原请求头（零开销，不加载插件）
└── 是
    ├── 加载插件（懒加载，仅首次）
    ├── 有已加载的插件？
    │   ├── 否 → 返回原请求头（占位符保持原样）
    │   └── 是
    │       ├── 依次调用每个插件的 resolve()
    │       ├── 首个成功解析的占位符生效（first-wins）
    │       ├── 至少有一个占位符被解析？
    │       │   ├── 否 → 返回原请求头
    │       │   └── 是 → 替换占位符，返回新请求头
    │       └── 插件 resolve() 抛错 → 记录警告，继续下一个插件
    └──
```

## 生命周期与配置

1. 在 Snow CLI 的设置界面中配置自定义请求头方案，在值中使用 `{{占位符}}` 语法。
2. 把插件文件放到 `~/.snow/plugin/custom_headers/` 下。
3. 启动（或重启）Snow CLI。
4. 后续所有 API 请求（chat / anthropic / gemini / responses / models）在发送前都会自动解析占位符。

> 占位符解析发生在每次 API 请求发送之前。如果插件的 `resolve()` 涉及网络请求（如获取 OAuth Token），建议在插件内部做缓存以避免每次请求都发起网络调用。

## 示例：时间戳与环境变量插件

下面是一个完整的示例插件，解析 `{{timestamp}}` 和 `{{env}}` 两个占位符：

```js
// ~/.snow/plugin/custom_headers/timestamp-env.mjs

export default {
	id: 'timestamp-env',
	name: 'Timestamp & Environment Resolver',
	enable: true,

	async resolve(placeholders, context) {
		const result = {};

		for (const name of placeholders) {
			switch (name) {
				case 'timestamp':
					// ISO 8601 格式的当前时间戳
					result.timestamp = new Date().toISOString();
					break;

				case 'env':
					// 根据工作目录推断环境
					if (context.cwd.includes('/production/')) {
						result.env = 'prod';
					} else if (context.cwd.includes('/staging/')) {
						result.env = 'staging';
					} else {
						result.env = process.env.NODE_ENV || 'dev';
					}
					break;

				case 'platform':
					result.platform = context.platform;
					break;
			}
		}

		return result;
	},
};
```

配置自定义请求头：

| 键               | 值              |
| ---------------- | --------------- |
| `X-Request-Time` | `{{timestamp}}` |
| `X-Environment`  | `{{env}}`       |
| `X-Platform`     | `{{platform}}`  |

请求发送时，这些值会被自动替换为实际内容。

## 示例：OAuth Token 动态获取（带缓存）

下面是一个更实用的示例，从 OAuth 端点获取访问令牌并缓存，避免每次请求都重新获取：

```js
// ~/.snow/plugin/custom_headers/oauth-token.mjs

// 简单的内存缓存
let cachedToken = null;
let cachedAt = 0;
const CACHE_TTL_MS = 50 * 60 * 1000; // 50 分钟（Token 通常 1 小时过期）

async function fetchOAuthToken() {
	const now = Date.now();

	// 缓存未过期，直接返回
	if (cachedToken && now - cachedAt < CACHE_TTL_MS) {
		return cachedToken;
	}

	// 从环境变量读取 OAuth 配置
	const tokenUrl = process.env.OAUTH_TOKEN_URL;
	const clientId = process.env.OAUTH_CLIENT_ID;
	const clientSecret = process.env.OAUTH_CLIENT_SECRET;

	if (!tokenUrl || !clientId || !clientSecret) {
		throw new Error('OAuth environment variables not configured');
	}

	const response = await fetch(tokenUrl, {
		method: 'POST',
		headers: {'Content-Type': 'application/x-www-form-urlencoded'},
		body: new URLSearchParams({
			grant_type: 'client_credentials',
			client_id: clientId,
			client_secret: clientSecret,
		}),
	});

	if (!response.ok) {
		throw new Error(`OAuth token request failed: ${response.status}`);
	}

	const data = await response.json();
	cachedToken = data.access_token;
	cachedAt = now;

	return cachedToken;
}

export default {
	id: 'oauth-token',
	name: 'OAuth Token Resolver',
	enable: true,

	async resolve(placeholders) {
		const result = {};

		if (placeholders.includes('oauth_token')) {
			try {
				result.oauth_token = await fetchOAuthToken();
			} catch (error) {
				// 解析失败时不写入 result，占位符保持原样
				// Snow CLI 会记录警告日志
				console.error('[oauth-token] Failed to fetch token:', error.message);
			}
		}

		return result;
	},
};
```

配置自定义请求头：

| 键              | 值                       |
| --------------- | ------------------------ |
| `Authorization` | `Bearer {{oauth_token}}` |

## 一个文件注册多个插件

可以从同一个文件中注册多个插件：

```js
export const customHeaderPlugins = [
	{
		id: 'timestamp',
		async resolve(placeholders) {
			const result = {};
			if (placeholders.includes('timestamp')) {
				result.timestamp = Date.now().toString();
			}
			return result;
		},
	},
	{
		id: 'uuid',
		async resolve(placeholders) {
			const result = {};
			if (placeholders.includes('uuid')) {
				result.uuid = crypto.randomUUID();
			}
			return result;
		},
	},
];
```

适合多个插件共享工具函数的场景。

## 编写自己的插件：检查清单

1. **`id` 要稳定且唯一**。用于日志标识，一旦发布请保持不变。
2. **只返回成功解析的占位符**。无法解析的占位符不要写入返回值，留给其他插件处理。
3. **`resolve()` 不要抛错**。如果获取值失败，直接省略该占位符即可。抛错会被 Snow CLI 捕获并记录警告，但不会中断请求。
4. **网络请求要做缓存**。`resolve()` 在每次 API 请求时都会被调用，如果不缓存，每次请求都会触发网络调用。
5. **不要依赖请求顺序**。多个插件的加载顺序按文件名排序，不要假设你的插件一定先于或后于其他插件执行。
6. **占位符名称要有语义**。使用 `{{oauth_token}}` 而非 `{{t}}`，避免与其他插件冲突。
7. **可以利用 `context`**。`context.cwd` 和 `context.platform` 可以帮助你根据当前环境返回不同的值。
8. **插件作为 Node.js 模块运行**。你可以 `import` 任何 Node.js 内置模块（如 `crypto`、`fs`、`os`），也可以使用 `process.env` 读取环境变量。

## 故障排查

- **占位符没有被替换，原样发送了。**

  - 确认插件文件在 `~/.snow/plugin/custom_headers/` 目录下。
  - 确认扩展名是 `.js` / `.mjs` / `.cjs`。
  - 检查 Snow CLI 日志中是否有 `[custom-headers] failed to load plugin` 或 `Custom header plugin resolve failed`，语法错误和解析异常都会被记录。
  - 确认导出的是包含 `{id, resolve}` 的纯对象——校验失败时加载器会输出 `did not export a valid CustomHeaderPlugin`。
  - 确认插件没有设 `enable: false`。

- **插件加载了但占位符仍然没替换。**

  - 检查 `resolve()` 返回的对象中是否包含了对应的占位符名称作为键。
  - 确认返回的值是非空字符串——空字符串会被忽略。
  - 确认占位符名称匹配：`{{token}}` 对应返回值的键是 `"token"`（会自动 trim 空格）。

- **我想临时停用某个插件。**

  - 在插件对象中设置 `enable: false`，无需删除文件。停用后该插件不会被调用。

- **没有安装任何插件时会有影响吗？**

  - 不会。如果请求头值中不包含 `{{占位符}}`，插件系统完全不介入，行为与之前一致。即使值中包含占位符但没有安装插件，占位符也会原样发送，不会产生错误。
